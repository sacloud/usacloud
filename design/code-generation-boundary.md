# 生成コードと手書きコードの境界

本ドキュメントでは、usacloud において手書きで管理すべきコードと、`make gen` などで自動生成されるコードの境界を明確にします。

## 基本方針

usacloud は「リソース・コマンドの定義は手書きし、機械的なボイラープレートはコード生成に任せる」方針を採用しています。

- 手書きコード: ドメインロジック、パラメータ構造、サービスとの対応関係である。
- 生成コード: フラグ定義、サービス呼び出し関数のレジストリ登録、import ブートストラップである。

## 手書きコード

### `pkg/commands/<platform>/<resource>/resource.go`

`core.Resource` の定義。リソース名、カテゴリ、対応する service（`iaas-service-go` / `webaccel-api-go` 等）の型などを記述します。

### `pkg/commands/<platform>/<resource>/<command>.go`

`core.Command` とそのパラメータ struct の定義。コマンド名、カテゴリ、`ParameterInitializer`、カスタム `Func`/`ValidateFunc` などを記述します。

### `pkg/commands/<platform>/<resource>/columns.go`

テーブル出力用の `output.ColumnDef` 定義。

### `pkg/commands/<platform>/labels.go`

`core.LabelsExtractors` への extractor 登録。ID/Name/Tags による引数解決や補完に必要です。プラットフォーム単位の汎用実装として提供されるため、リソースごとに個別に作成する必要はありません。

### `pkg/commands/<platform>/resources.go`

プラットフォーム内の全リソースを `core.Resources` として列挙します。

### `pkg/commands/<platform>/category/*.go`

リソースカテゴリの定義。

### `pkg/vdef/definitions.go`

`options=xxx` タグで参照される値定義。

### `pkg/core/*`

コマンド実行フレームワークそのもの。コード生成の対象外です。

## 生成コード

以下のファイルは `tools/gen-commands` によって生成されます。原則として手で編集しません。

### `pkg/commands/<platform>/<resource>/zz_*_gen.go`

各コマンドのパラメータ struct に対する Cobra/Pflag 用メソッド群を生成します。

生成される内容:

- `CleanupEmptyValue(fs *pflag.FlagSet)` ... 未指定のポインタフィールドをnilに戻す。
- `buildFlags(fs *pflag.FlagSet)` ... 各フィールドに対応するフラグを登録する。
- `normalizeFlagName(...)` ... エイリアスを正規化する。
- `buildFlagsUsage(cmd *cobra.Command)` ... カテゴリ別のフラグヘルプを構築する。
- `setCompletionFunc(cmd *cobra.Command)` ... `options`タグ付きフィールドの補完関数を登録する。
- `SetupCobraCommandFlags(cmd *cobra.Command)` ... 上記をまとめて呼び出す。

### `pkg/services/<platform>/<resource>_services_gen.go`

service（`iaas-service-go` / `webaccel-api-go` 等）のメソッドを呼び出す関数を `pkg/services/registry` に登録します。

生成される内容:

- `registry.SetDefaultServiceFunc(...)` ... コマンド名に対応するサービスメソッド呼び出しである。
- `registry.SetDefaultListAllFunc(...)` ... リソース解決/補完用の全件検索である。

`Command.Func` が明示的に設定されている場合、対応する `SetDefaultServiceFunc` は生成されません。

### `pkg/commands/<platform>/services_gen.go`

生成済みサービスパッケージを import するだけのブートストラップファイルです。

## 生成のトリガー

コード生成は `go generate ./...` 経由で実行されます。`GNUmakefile` では以下のターゲットが定義されています。

```makefile
gen: _gen set-license fmt goimports

_gen:
    go generate ./...
```

`pkg/resources.go` に `//go:generate go run github.com/sacloud/usacloud/tools/gen-commands/` があるため、`go generate ./...` 実行時に `tools/gen-commands` が起動します。

## 生成コードの特徴

生成ファイルには以下のコメントヘッダーが付きます。

```go
// Code generated by 'github.com/sacloud/usacloud/tools/gen-commands'; DO NOT EDIT
```

これらのファイルは Git で管理対象としますが、編集は `tools/gen-commands` のテンプレートや `tools/*.go` を変更して再生成するのが正しい方法です。

## 編集すべきでないファイルへの誤った変更を防ぐための指針

以下のファイルを直接編集しようとしている場合、おそらく境界を誤解しています。

| 誤った変更 | 正しい対応 |
| --- | --- |
| `zz_*_gen.go` のフラグ定義を直接修正 | パラメータ struct の `cli` タグを修正し再生成 |
| `*_services_gen.go` の service 呼び出しを直接修正 | `Command.Func` を手書きで設定するか、service 側を修正 |
| 生成ファイルの import を手動で整理 | `make goimports` または `make gen` を実行 |

## 生成コードが壊れた場合のリカバリ

生成コードと手書きコードの整合性が取れなくなり、ビルドが通らなくなった場合、以下を試してください。

```sh
make clean-all  # 生成ファイルを全削除
make gen        # 再生成
```

`clean-all` は `*_gen.go` を削除するため、手書きファイルに影響を与えません。

## まとめ

| 種別 | 例 | 管理方法 |
| --- | --- | --- |
| 手書き | `resource.go`, `<command>.go`, `columns.go`, `labels.go` | 手動で編集 |
| 生成 | `zz_*_gen.go`, `*_services_gen.go`, `services_gen.go` | `make gen` で生成 |
| フレームワーク | `pkg/core/*` | 手動で編集（生成対象外） |

この境界を守ることで、新しいリソース・コマンドの追加がフレームワークの変更なしに機械的に行えるようになります。
